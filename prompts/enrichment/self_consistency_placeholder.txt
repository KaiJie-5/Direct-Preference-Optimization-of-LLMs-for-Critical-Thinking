CAT-CoT structure:

1. Segment meaning
- What is the participant saying in the target segment?
- What specific words or phrases in the segment support this interpretation?
- Is previous_context or next_context needed to understand the segment?
- Do not interpret the whole interview as one meaning unit.

2. Research-question relevance
- There may be one supplied research question or multiple supplied research questions.
- Identify which supplied research question(s), if any, this segment helps answer.
- If the segment only gives background, description, or unclear material, say so.
- Treat research-question relevance as the main test of analytical usefulness.
- Do not say vaguely that something answers "the research questions"; specify which one(s) or say that none are clearly answered.

3. Candidate-code check
- Decide whether any candidate example code fits the segment.
- Do not force a candidate code.
- If candidate codes miss the specific meaning of the segment, consider a provisional new code.
- Candidate codes are guidance only, not already-applied human codes.

4. Category A: plausible but unsupported wrong code
- Create exactly one wrong code.
- It must sound plausible for the wider dataset, but it must not be supported by this target segment.
- Do not create a random nonsense code.
- Use a quote from the target segment to show what the participant actually says.
- Explain why that quote does not support the wrong code.

5. Category B: descriptive but not research-question useful code
- Create exactly one descriptive code.
- It must be true of the target segment at a surface level.
- It must fail because it does not help answer the supplied research question(s).
- This category is about poor research-question relevance, not broadness.
- Test: if this code were removed from the analysis, would the analysis lose anything important for answering the research question(s)? If not, it belongs here.

6. Category C: too broad code
- Create exactly one too-broad code.
- It must be related to the segment and at least partly relevant to the supplied research question(s).
- It must fail because it hides the specific experience, interaction, barrier, tension, motivation, or mechanism in the participant's words.
- This category is on-aim but too vague.
- State what specific meaning is lost by using this broad label.

7. Category D: useful analytical code
- Create exactly one useful analytical code.
- It must be grounded in the target segment.
- It must help answer at least one supplied research question.
- It must be specific enough to distinguish this segment from many other segments.
- It should capture an experience, interaction, barrier, challenge, tension, motivation, mechanism, or service/design implication.
- The label must be an insight-like phrase or short sentence.
- Do not use a single noun, vague topic, or whole research question as the label.

8. Contrastive judgement
- Compare the four codes directly.
- Explain why the wrong code is unsupported.
- Explain why the descriptive code is true but analytically weak.
- Explain why the too-broad code is relevant but loses important meaning.
- Explain why the useful analytical code is better than the other three.
- Explain which supplied research question(s) the useful code helps answer.

9. Reflective questions
- Create exactly 3 reflective questions.
- Each question must test the useful analytical code.
- At least one question must check whether the interpretation over-reads the quote.
- At least one question must check whether another interpretation is possible.
- At least one question must check what context a human researcher would need before trusting the code.
